垃圾回收器的设计需要考虑三个问题：哪些内存需要回收？什么时候回收？如何回收？
### 1 哪些内存需要回收？
Java内存运行时区域的各个部分，其中程序计数器、虚拟机栈、本地方法栈3个区域随线程而生，随线程而灭，栈中的栈帧随着方法的进入和退出而有条不紊地执行着出栈和入栈操作。每一个栈帧中分配多少内存基本上是在类结构确定下来时就已知的，因此这几个区域的内存分配和回收都具备确定性， 在这几个区域内就不需要过多考虑如何回收的问题，当方法结束或者线程结束时，内存自然就跟随着回收了。
Java **堆和方法区**这两个区域则有着很显著的不确定性: 一个接口的多个实现类需要的内存可能会不一样，一个方法所执行的不同条件分支所需要的内存也可能不一样，只有处于运行期间，我们才能知道程序究竟会创建哪些对象，创建多少个对象，这部分内存的分配和回收是动态的。垃圾收集器所关注的正是这部分内存该如何管理。
### 2 什么时候回收？
#### 1.1 Minor GC（新生代 GC）
- **Eden 区空间不足**：新生代主要由 Eden 区和两个 Survivor 区组成。当新创建的对象在 Eden 区分配内存时，如果 Eden 区没有足够的空间来分配新对象，就会触发 Minor GC。
#### 1.2 Major GC / Full GC（老年代 GC）
- **新生代对象晋升到老年代导致老年代空间不足**：在 Minor GC 过程中，一些存活时间较长的对象会从新生代晋升到老年代。如果老年代没有足够的空间来容纳这些晋升的对象，就会触发 Major GC 或 Full GC。
- **显式调用 `System.gc()` 方法**：在 Java 代码中，可以显式调用 `System.gc()` 方法来建议 JVM 进行垃圾回收。不过，这只是一个建议，JVM 并不一定会立即执行 Full GC，具体是否执行由 JVM 决定。
- **永久代（JDK 7 及以前）或元空间（JDK 8 及以后）空间不足**：在 JDK 7 及以前，永久代用于存储类的元数据等信息；在 JDK 8 及以后，永久代被元空间取代。如果永久代或元空间的空间不足，会触发 Full GC。
- **CMS GC 出现并发模式失败**：CMS（Concurrent Mark Sweep）是一种以获取最短回收停顿时间为目标的垃圾收集器。在 CMS 并发标记阶段，如果有新的对象进入老年代，而老年代没有足够的空间，就会导致并发模式失败，从而触发 Full GC。
#### 1.3 Mixed GC（混合 GC）
- **达到 G1 收集器的堆占用阈值**：G1（Garbage First）收集器会根据堆的使用情况，在达到一定的堆占用阈值时触发 Mixed GC。
  这个阈值可以通过 `-XX:InitiatingHeapOccupancyPercent` 参数进行配置，默认值是 45。
- **新生代空间不足**：当新生代空间不足时，G1 收集器也会触发 Mixed GC，同时回收新生代和部分老年代的垃圾对象。
